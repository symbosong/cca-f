## 领域 1：智能体架构与编排

### 任务 1.1：设计和实现自主任务执行的智能体循环

**Q1.** 生产数据显示，在 12% 的案例中，你的智能体完全跳过 `get_customer`，仅使用客户口述的姓名直接调用 `lookup_order`，偶尔导致账户识别错误和退款失误。什么改变能最有效地解决这个可靠性问题？

A) 添加程序化前置条件，在 `get_customer` 返回已验证的客户 ID 之前，阻止 `lookup_order` 和 `process_refund` 的调用。  
B) 增强系统提示，声明在任何订单操作之前必须通过 `get_customer` 进行客户验证。  
C) 添加少样本示例，展示智能体即使在客户主动提供订单详情时也始终先调用 `get_customer`。  
D) 实现路由分类器，分析每个请求并仅启用适合该请求类型的工具子集。  

**正确答案：A**
当关键业务逻辑需要特定的工具调用顺序时，程序化强制执行提供了提示方法无法实现的确定性保证。选项 B 和 C 依赖于概率性的 LLM 合规性。选项 D 解决的是工具可用性而非工具排序问题。

**Q13.** 你正在使用 Claude Agent SDK 构建客户支持智能体。该智能体通过调用 `lookup_order`、`get_invoice` 和 `process_refund` 等工具处理账单争议。代码审查发现，在每次工具调用后，你的循环检查 Claude 的响应文本是否包含"I have completed"这个短语来决定是否停止。

这种循环终止方法的主要问题是什么？

A) 智能体永远不会终止，因为 `process_refund` 总是返回成功消息，阻止"I have completed"出现。  
B) 依赖助手文本中的自然语言信号是不可靠的；正确的方法是检查 `stop_reason`，只有当其等于 `"end_turn"` 时才终止。  
C) 循环应该在任何工具调用失败时立即终止，因为失败后继续会破坏对话历史。  
D) 检查响应文本仅在同步模式下有效；在异步上下文中必须使用回调处理器进行正确的循环控制。  

**正确答案：B**
检查 `stop_reason` 是智能体循环控制的标准方法：当 `stop_reason` 为 `"tool_use"` 时继续迭代，当其为 `"end_turn"` 时停止。"I have completed"等自然语言短语不是可靠的终止信号，因为 Claude 的措辞在不同响应和提示更新中会有变化。选项 A 和 C 基于对工具与响应文本交互方式的错误假设。选项 D 混淆了异步执行模式和循环控制逻辑。

**Q14.** 一个开发者生产力智能体通过调用 `search_codebase`、`read_file` 和 `run_tests` 来调查 bug 报告。在代码审查中，同事提议添加迭代上限：如果循环在 15 次工具调用后仍未完成，则以通用的"调查未完成"响应终止。

对这种设计的正确评价是什么？

A) 迭代上限是推荐的主要停止机制，因为它们可以防止生产环境中的失控成本。  
B) 迭代上限对于长时间运行的任务是合理的安全边界，但不应作为主要终止机制；`stop_reason: "end_turn"` 仍然是权威信号。  
C) 迭代上限应替换为基于时间的超时，因为 Token 计数比迭代计数更能可靠地衡量完成度。  
D) 如果系统提示指示智能体始终只请求最少需要的工具，则迭代上限是不必要的。  

**正确答案：B**
智能体循环应主要在 `stop_reason` 等于 `"end_turn"` 时终止，这表示模型已完成任务。迭代上限可以作为防止循环失控的安全护栏，但将其作为主要停止机制意味着循环可能会截断合法的多步调查。选项 A 和 C 错误描述了推荐做法。选项 D 混淆了基于提示的指导与循环控制：即使是精心提示的智能体也可能合法地需要多次工具调用来处理复杂调查。

**Q15.** 你的多智能体研究系统的智能体循环在发送下一个请求之前将每个工具结果追加到对话历史中。团队成员建议改为将所有工具结果存储在单独的数据库中，每次迭代只向模型提供摘要而非完整结果。

在什么条件下，这种改变最可能降低智能体性能？

A) 当工具结果包含图像或文件附件等二进制数据时。  
B) 当模型需要同时推理多个工具结果以确定下一步操作时，因为摘要可能遗漏推理所需的细节。  
C) 当每个会话的工具调用次数超过 10 次时，因为更大的历史记录会减慢 API 速度。  
D) 当工具返回结果的速度快于 200ms 时，使历史追加变得多余。  

**正确答案：B**
智能体循环依赖于工具结果存在于对话历史中，以便模型在决定下一步操作之前推理它已经发现的内容。摘要可能遗漏模型做出正确决策所需的字段值、错误代码或条件数据。选项 A 是一个有效的关注点，但不是基于文本的研究任务的主要性能风险。选项 C 和 D 不是此架构的已知故障模式。

### 任务 1.2：使用协调器-子智能体模式编排多智能体系统

**Q7.** 运行"AI 对创意产业的影响"任务后，每个子智能体都成功完成，但最终报告只涵盖了视觉艺术，缺少音乐、写作和电影制作。协调器将主题分解为："AI 在数字艺术创作中"、"AI 在图形设计中"和"AI 在摄影中"。最可能的根本原因是什么？

A) 综合智能体缺少识别其从其他智能体收到的发现中的覆盖缺口的指令。  
B) 协调器智能体的任务分解过于狭窄，导致子智能体的分配未能覆盖所有相关领域。  
C) 网络搜索智能体的查询不够全面，需要扩展以覆盖更多创意产业领域。  
D) 文档分析智能体由于过于严格的相关性标准，过滤掉了与非视觉创意产业相关的来源。  

**正确答案：B**
协调器的日志直接揭示了根本原因：它将"创意产业"仅分解为视觉艺术子任务。子智能体正确执行了分配给它们的任务。问题在于它们被分配了什么任务。

**Q16.** 你的多智能体研究系统使用一个协调器，无论查询复杂度如何，都始终将每个查询路由到所有四个子智能体（网络搜索、文档分析、综合和报告生成）。对于像"Anthropic API 是哪年发布的？"这样的简单事实问题，系统需要 45 秒并产生不必要的成本。

什么设计更改最直接地解决这个问题？

A) 用静态路由表替换协调器，将查询关键词映射到特定的子智能体流水线。  
B) 让每个子智能体评估自身与当前查询的相关性，自行选择加入或退出流水线。  
C) 设计协调器根据查询复杂度和类型动态选择要调用的子智能体，而不是始终通过完整流水线路由。  
D) 通过将网络搜索和文档分析合并为单个"检索"智能体，将子智能体数量从四个减少到两个。  

**正确答案：C**
一个无论查询复杂度如何都始终运行完整流水线的协调器会浪费资源并为简单请求增加延迟。正确的设计是协调器评估查询并仅选择该特定任务所需的子智能体。选项 A 用脆弱的关键词表替代了自适应智能。选项 B 将协调逻辑分散到子智能体中，破坏了中心辐射模式并降低了可观察性。选项 D 不必要地降低了能力，而不是改进路由逻辑。

**Q17.** 你的研究系统中的协调器智能体将文档分析委托给了一个子智能体。子智能体完成后，协调器注意到发现不完整：子智能体只覆盖了五个指定来源中的三个。协调器需要分析剩余的两个来源。

重新委托这项工作的正确方法是什么？

A) 使用部分发现调用综合智能体，指示其根据三个已完成分析的模式推断缺失来源可能包含的内容。  
B) 协调器应使用明确指定仅两个缺失来源的提示重新调用文档分析子智能体，包括之前完成的发现作为上下文。  
C) 将所有五个来源再次发送给新的文档分析子智能体实例，重新分析已完成的三个来源以及两个缺失的来源。  
D) 让协调器直接生成对剩余两个来源的分析，而不是生成另一个子智能体委托。  

**正确答案：B**
协调器的职责包括评估输出的缺口并重新委托针对性工作以弥补这些缺口。仅使用未完成的来源重新调用子智能体，并将已完成的发现作为上下文提供，既高效又正确。选项 A 要求综合智能体编造内容，这会损害报告准确性。选项 C 浪费资源重新处理已完成的来源。选项 D 绕过了协调器-子智能体架构，忽略了子智能体的专业化。

### 任务 1.3：配置子智能体调用、上下文传递和生成

**Q18.** 你正在构建一个使用 `Task` 工具将研究任务委托给子智能体的协调器。测试系统时，协调器无法调用任何子智能体。查看协调器的 `AgentDefinition`，你发现 `allowedTools` 字段设置为 `["web_search", "read_document"]`。

最可能的失败原因是什么？

A) `Task` 工具需要在调用前进行显式的 `subagent_endpoint` 配置。  
B) `"Task"` 未包含在协调器的 `allowedTools` 中，因此它无法生成子智能体。  
C) 协调器的系统提示未包含使用 `Task` 工具的指令，因此模型从未尝试调用它。

**正确答案：B**

### 任务 4.6：设计多实例和多遍审查架构

**正确答案：A**
将审查拆分为聚焦的遍历直接解决了注意力分散问题。选项 B 将负担转移给开发者而没有改进系统。选项 C 误解了更大的上下文窗口不能解决注意力质量问题。选项 D 通过要求对可能仅间歇性被发现的问题达成共识来抑制真实 bug 的检测。

**Q66.** 一位开发者在一个会话中使用 Claude Code 生成了 300 行模块，然后要求同一会话审查代码中的 bug。审查返回"看起来不错"，仅有少量风格建议，但遗漏了同事在手动审查中发现的两个逻辑错误。最可能的原因是什么，什么架构更改可以解决？

A) 上下文窗口太大；修复方法是将代码生成限制在更小的块中，以便审查需要处理的内容更少。  
B) 审查实例保留了生成过程的推理上下文，使其不太可能质疑自己的决策。使用没有生成上下文的第二个独立 Claude 实例进行审查。  
C) 审查提示太模糊；在同一会话中添加更明确的审查标准可以发现遗漏的错误。  
D) 用于生成的模型层级比用于审查的更强大；将两者切换到同一层级可解决质量差距。  

**正确答案：B**
自我审查限制是一个已知的模式：当模型保留代码生成的推理上下文时，它不太可能识别自己输出中的错误。没有生成器上下文的独立审查实例以没有先入为主假设的方式审查代码，能发现生成实例会忽略的问题。选项 A 解决的是上下文长度，而非推理上下文偏差。选项 C 可能改善某些类别问题的检测但不解决根本的自我审查限制。选项 D 解决的是模型选择，而非审查自己输出的结构性问题。

**Q68.** 一位开发者构建了代码生成流水线，Claude 生成 200 行模块，然后同一会话审查生成的代码的正确性。审查仅返回少量风格建议，遗漏了错误处理路径中的逻辑错误。第二位开发者手动审查代码时立即发现了 bug。什么架构更改可以使审查阶段更有效？

A) 在生成会话的系统提示中添加更明确的审查标准以帮助其捕获逻辑错误。  
B) 使用没有生成上下文的第二个独立 Claude 实例进行审查，因为生成实例保留了推理上下文，使其不太可能质疑自己的决策。  
C) 在生成后立即运行审查，在任何其他工具调用累积到上下文窗口之前。  
D) 为审查步骤切换到更大的模型层级以提高推理质量。  

**正确答案：B**
当模型审查在同一会话中生成的代码时，它保留了生成过程的推理上下文，不太可能识别自己输出中的错误。独立的审查实例以没有先入为主假设的方式审查代码，能发现生成实例忽略的问题。这是多实例审查架构的文档化理由。选项 A 改善了审查提示但没有解决根本的自我审查限制。选项 C 管理上下文长度但没有移除生成器保留的推理偏差。选项 D 提高了通用能力但没有改变模型审查自己工作的结构性问题。

## 领域 5：上下文管理与可靠性

### 任务 5.1：管理对话上下文以在长交互中保留关键信息

**Q69.** 一个客户支持智能体正在处理跨越 20+ 轮的复杂账单争议。客户在对话早期提到他们在 3 月 3 日因 2 月 28 日已取消的服务被收取了 847.50 美元。在对话中途，智能体将此费用引用为"上个月的多收费"而没有具体金额。客户对智能体的案件理解提出异议。什么上下文管理技术能防止这种情况？

A) 在每次对话开始时使用 `/compact` 将先前的轮次总结为更短的表示。  
B) 将交易事实（金额、日期、订单号、状态）提取到持久的"案件事实"块中，在对话中每个后续提示的开头包含。  
C) 通过切换到更大的层级增加模型的上下文窗口，以在不进行总结的情况下保留完整对话。  
D) 指示智能体在每次响应前重新阅读完整的对话历史。  

**正确答案：B**
将精确的交易事实提取到持久的"案件事实"块中，确保特定的数值、日期和客户声明的细节在每个提示轮次中显式可用，而非埋没在增长的对话历史中。渐进式总结有风险将这些细节压缩为模糊的引用如"上个月的多收费"。选项 A 的压缩对上下文长度管理有用但可能丢失交易事实的精度。选项 C 不能防止渐进式总结且不是可靠的解决方案。选项 D 对长对话不实际，仍然不能防止事实随着上下文增长而丢失。

**Q70.** 一个多智能体研究流水线聚合来自六个子智能体的发现，每个返回 800-1,200 Token 的原始工具输出和推理。综合智能体在单个大消息中接收所有结果。最终报告持续遗漏或矛盾出现在聚合输入中间部分的发现。最可能的原因和最有效的缓解措施是什么？

A) 子智能体返回了冲突信息；在综合之前添加去重步骤。  
B) 综合智能体达到了输出 Token 限制；增加 `max_tokens` 以允许更长的响应。  
C) "中间丢失"效应导致模型可靠地处理长输入开头和结尾的内容但遗漏中间部分。通过将关键发现摘要放在聚合输入的开头并使用显式标题组织部分来缓解。  
D) 综合智能体的上下文窗口已耗尽；将部分子智能体输出通过二次总结智能体路由后再传递给综合。  

**正确答案：C**
"中间丢失"效应是一个有据可查的限制，模型对长输入中间的内容注意力不如开头和结尾可靠。将关键发现放在开头并使用显式部分标题组织聚合输入可显著缓解此效应。选项 A 解决数据一致性但不解决位置注意力问题。选项 B 解决输出长度而非输入注意力。选项 D 增加了流水线复杂性，可能有助于上下文长度，但不能单独解决位置注意力问题。

### 任务 5.2：设计有效的升级和歧义解决模式

**Q3.** 你的智能体首次联系解决率为 55%，远低于 80% 的目标。日志显示它将简单案例升级，同时尝试自主处理需要政策例外的复杂情况。改善升级校准最有效的方法是什么？

A) 在系统提示中添加明确的升级标准，配合少样本示例展示何时升级与何时自主解决。  
B) 让智能体在每次响应前自我报告置信度分数（1-10），当置信度低于阈值时自动将请求路由给人工。  
C) 部署一个在历史工单上训练的独立分类器模型，在主智能体开始处理之前预测哪些请求需要升级。  
D) 实现情感分析以检测客户挫败感水平，当负面情感超过阈值时自动升级。  

**正确答案：A**
添加明确的升级标准配合少样本示例直接解决了根本原因：不清晰的决策边界。选项 B 失败是因为 LLM 自我报告的置信度校准很差。选项 C 过度工程化。选项 D 完全解决的是不同的问题。

**Q71.** 客户打开支持聊天说"我想和人工客服交谈。我已经处理这个账单问题三周了，我很沮丧。" 智能体已经识别这是一个标准的账单调整，可以使用其可用工具在 2 步内解决。你的智能体回应说"我可以立即帮您处理，让我调出您的账户。"并继续调查。客户再次要求人工。这种行为有什么问题，应该如何纠正？

A) 智能体应在第一次请求时立即尊重客户对人工的明确请求，不尝试先进行调查。  
B) 智能体应检测负面情感，当挫败感分数超过定义的阈值时升级。  
C) 智能体应完成调查，提出建议的解决方案，仅在客户仍然坚持后才升级。  
D) 智能体应为等待道歉，然后继续自动解决，因为案例在其能力范围内。  

**正确答案：A**
当客户明确请求人工时，无论智能体是否认为自己能解决问题，该请求都必须立即被尊重。在明确的升级请求后继续调查违反了任务 5.2 中记录的设计原则。选项 B 引入情感评分作为代理，但客户已经表达了明确的偏好。情感分析在此处无关。选项 C 在调查进行时延迟尊重请求，进一步加剧客户的挫败感。选项 D 是智能体当前的错误行为：当客户提出明确请求时，能力不是决定因素。

### 任务 5.3：在多智能体系统中实现错误传播策略

**Q72.** 你的网络搜索子智能体在获取竞争对手分析的结果时遇到超时。工程团队实施更改，使子智能体在内部捕获超时并返回带有 `status: "success"` 和 `results: []` 的空结果集。协调器收到此结果，综合智能体生成了一份完全缺少竞争分析部分的报告，没有任何异常指示。这种方法有什么问题？

A) 静默返回空的成功结果阻止了协调器做出任何恢复决策，并允许不完整的工作作为完整输出通过。  
B) 综合智能体应该负责检测空部分并直接重新触发搜索子智能体。  
C) 协调器应始终在继续综合之前验证每个子智能体返回了非空结果。  
D) 应增加超时阈值，以便子智能体在返回真实结果之前不会超时。  

**正确答案：A**
通过将空结果作为成功返回来静默抑制错误是一个明确记录的反模式。它移除了协调器恢复、重试或用覆盖缺口标注输出的能力。选项 B 错误地将恢复责任转移到综合智能体，它缺乏重新运行搜索的工具和上下文。选项 C 添加了协调器级别的验证作为保障，但没有解决根本原因：子智能体将失败错误表示为成功。选项 D 解决的是症状（超时阈值）而非错误传播设计缺陷。

### 任务 5.4：在大型代码库探索中有效管理上下文

**Q73.** 一个智能体正在对 200,000 行旧版代码库进行深度探索以理解其支付处理流程。在大约 90 分钟和数十次文件读取后，智能体开始引用"支付系统中的标准模式"而非它之前发现的特定类和流程，其回答与第一个小时的发现变得不一致。最可能的原因和最佳缓解策略是什么？

A) 智能体已达到速率限制；修复方法是在工具调用之间添加延迟以保持在限制内。  
B) 上下文退化：随着会话增长，会话早期的特定发现与通用知识和智能体的位置-注意力效应竞争。通过让智能体在整个会话中维护一个记录关键发现的便笺文件来缓解。  
C) 旧版代码库的文件结构对单个智能体来说太复杂；将其分解为独立子系统并同时在每个子系统上运行单独的智能体。  
D) 智能体使用了错误的工具；将文件探索从 `Read` 切换到 `Grep` 可以减少上下文消耗。  

**正确答案：B**
扩展会话中的上下文退化是一个已知模式：模型开始给出不一致的答案并引用通用知识而非之前发现的特定发现。让智能体维护记录关键发现的便笺文件创建了一个持久的、显式的记录，可以在后续提示中引用和包含，防止特定细节因上下文压力而丢失。选项 A 错误地将原因识别为速率限制。选项 C 增加了并行性但没有解决单会话退化问题。选项 D 更改了工具但没有解决上下文增长或退化。

**Q74.** 你正在使用 Claude Agent SDK 运行多阶段代码库调查。在 45 分钟的探索（读取文件、跟踪调用链、构建依赖图）后，你进入第二阶段：识别认证模块中的安全漏洞。你注意到智能体现在将认证模块描述为"使用标准 JWT 模式"，而在会话早期它发现了自定义 Token 签名实现。什么技术最有效地在这些阶段边界之间保留关键发现？

A) 让智能体维护一个便笺文件，在每个阶段后记录关键发现，并在后续阶段开始时引用该文件而非依赖对话历史。  
B) 在开始第二阶段前使用 `/compact` 释放上下文空间，允许智能体根据需要重新读取文件。  
C) 在每个阶段开始时用新的上下文重新启动会话，传递你发现内容的手动摘要。  
D) 增加 `max_tokens` 参数，以便模型可以持有更多上下文而不压缩早期发现。  

**正确答案：A**
便笺文件是在上下文退化明显时跨上下文边界持久化关键发现的推荐技术。智能体可以在探索期间写入结构化笔记并在阶段边界读回它们，确保关键发现（如自定义 Token 签名实现）不会因上下文压缩而丢失。选项 B 使用 `/compact` 释放空间，但这压缩了对话历史，可能丢失关于自定义 Token 签名的特定发现。选项 C 是有效但代价高的方法：重新启动会话丢失了累积的理解并需要手动干预。选项 D 不正确因为 `max_tokens` 控制输出长度而非上下文窗口大小，不能防止退化。

### 任务 5.5：设计人工审查工作流和置信度校准

**Q75.** 一个保险理赔的结构化数据提取系统在验证集上达到 97% 的整体准确率。团队提议在没有人工审查的情况下自动化完整工作流。同事认为这个聚合指标掩盖了重要风险。最重要的关注点是什么？

A) 97% 的准确率意味着 3% 的理赔会有错误，如果不加检查可能造成法律责任。  
B) 聚合准确率指标可以掩盖特定文档类型或字段的糟糕表现。一个准确率为 85% 的理赔类型或字段类别会被其他地方的强表现所隐藏，在自动化之前需要分层分析。  
C) 验证集可能不能代表生产数据量，因此 97% 的数字不可信。  
D) 无论准确率指标如何，都应始终保留人工审查，因为保险决策的自动化会产生监管风险。  

**正确答案：B**
核心风险是聚合指标可以隐藏特定细分市场的糟糕表现。在关键字段如 `claim_amount` 上准确率为 85% 的文档类型会被其他文档类型的强表现所掩盖。在自动化之前，应按文档类型和字段细分验证准确率，以确认所有维度上的一致表现。选项 A 是有效的关注点但描述的是错误率而非掩盖风险。选项 C 提出数据质量关注但不是关于特定 97% 数字最重要的关注点。选项 D 提出政策论点但没有涉及识别的分析差距。

**Q76.** 一个提取流水线平均准确率很高，但团队想实现基于置信度的路由：高置信度提取自动进行，低置信度的交给人工审查。一位开发者提议直接使用模型声称的置信度分数作为路由阈值。这种方法的关键缺陷是什么？

A) 置信度分数为每个响应增加 Token，不必要地增加 API 成本。  
B) 模型置信度分数必须针对标注的验证集进行校准，以确定什么分数阈值实际对应可接受的准确率。未校准的原始分数不是提取正确性的可靠预测器。  
C) 置信度分数仅在模型输出单个提取时有效；多字段文档需要不同的方法。  
D) 基于置信度分数的路由创建了两条独立的代码路径，比单一统一的审查工作流更难维护。  

**正确答案：B**
模型的原始置信度分数需要校准：如果不将声称的置信度与标注验证集上的实际正确性进行比较，就没有"0.85"分数与可接受错误率之间的可靠映射。校准阈值确保路由截止点对应已知的准确率水平。选项 A 是次要的操作关注，不是关键缺陷。选项 C 不正确；置信度分数可以在多字段文档的字段级别应用。选项 D 是工程权衡，不是置信度方法本身的缺陷。

### 任务 5.6：Preserve information provenance and handle uncertainty in multi-source synthesis

**Q77.** 一个研究综合智能体正在合并来自五个搜索不同来源的子智能体的发现。最终报告声称"2024 年医疗保健领域的 AI 采用率达到 45%"但未引用此数据来自哪个来源。审查者无法验证该声明，因为子智能体的个别输出未被保留。对流水线的什么结构性更改能防止这种来源丢失？

A) 要求综合智能体在报告末尾添加通用的"咨询的来源"部分，列出所有访问的来源。  
B) 要求子智能体输出结构化的声明-来源映射（声明文本、来源 URL、文档名称、相关摘录），并指示综合智能体保留和合并这些映射到最终报告中，而非将它们总结掉。  
C) 将所有原始子智能体输出存储在单独的日志文件中，以便在需要验证声明时查阅。  
D) 添加综合后审查步骤，由单独的智能体将报告中的每个声明与原始子智能体输出进行核对。  

**正确答案：B**
来源丢失的根本原因是总结步骤在不保留声明-来源映射的情况下压缩了发现。要求子智能体输出结构化映射并指示综合智能体保留和合并它们，确保最终报告中的每个声明都通过流水线携带其来源归属。选项 A 生成已咨询来源的列表但不将个别声明与特定来源关联。选项 C 存储原始数据作为后备但不将来源集成到报告本身中。选项 D 在事后添加验证步骤但不能防止结构性来源丢失。

### 样题 1：程序化强制 vs 提示引导 〔领域 1 · 任务 1.4〕

> **场景**：生产数据显示，12% 的案例中智能体跳过 `get_customer` 验证直接调用 `lookup_order`，导致错误退款。什么变更最有效？

A) 添加程序化前置条件，阻止退款直到客户验证完成  
B) 增强系统提示，声明客户验证是强制性的  
C) 添加少样本示例，展示总是先调用 `get_customer`  
D) 实现路由分类器，仅启用适合该请求类型的工具子集  

> **答案 A**
>
> **解题思路**：题目问"最有效"地解决一个发生在 12% 案例中的可靠性问题，且涉及财务操作（退款）。
>
> - A **程序化前置条件**提供确定性保证——代码级阻止，0% 违规率
> - B ❌ 提示指令是概率性合规——"大部分时候遵守"对退款业务不够
> - C ❌ 少样本示例同理，改善但不能保证
> - D ❌ 路由分类器解决的是"选哪个工具"而非"工具调用顺序"
>
> **核心原则**：确定性需求 → 程序化强制；软性偏好 → 提示引导

### 样题 2：工具描述优化 〔领域 2 · 任务 2.1〕

> **场景**：用户问订单信息时，智能体频繁调用 `get_customer` 而非 `lookup_order`。两个工具描述都很简约。改善工具选择的"第一步"？

A) 添加 5-8 个少样本示例展示正确的工具选择模式  
B) 扩展每个工具描述，包含输入格式、示例查询和边界说明  
C) 实现基于关键词的路由层，在每轮前预选合适的工具  
D) 合并为单个 `lookup_entity` 工具  

> **答案 B**
>
> **解题思路**：关键词是"第一步"——要求低成本、高杠杆的改动。
>
> - B 工具描述是 LLM 选择工具的**主要机制**，改描述是成本最低、效果最大的第一步
> - A ❌ 少样本示例增加 Token 开销，不解决根本问题（描述不清）
> - C ❌ 路由层是过度工程化（还没试过改描述就上分类器）
> - D ❌ 合并工具是大改动，不适合"第一步"
>
> **核心原则**：先改描述（低成本高杠杆）→ 再加示例 → 最后上工程方案

### 样题 3：升级校准 〔领域 5 · 任务 5.2〕

> **场景**：客户支持智能体升级决策不一致——升级简单案例，却自主处理需要策略例外的复杂情况。如何改进？

A) 添加带少样本示例的明确升级标准  
B) 让智能体自我报告置信度分数  
C) 部署独立情感分类器  
D) 情绪分析检测挫折度自动升级  

> **答案 A**
>
> **解题思路**：根因是"不清晰的决策边界"——智能体不知道什么该升级、什么不该。
>
> - A 明确标准 + 具体示例 = 可预测的决策边界
> - B ❌ LLM 自我报告的置信度**校准很差**（已知问题），不可靠
> - C ❌ 还没试过改提示就上 ML 基础设施 = 过度工程化
> - D ❌ 情绪与案例复杂性不相关——愤怒的客户可能有简单问题，平静的客户可能有严重问题
>
> **核心原则**：先明确标准 + 示例（低成本）→ 再上工程方案

### 样题 4：项目命令位置 〔领域 3 · 任务 3.2〕

> **场景**：创建 `/review` 命令让团队克隆后自动可用。文件放哪里？

A) 项目仓库的 `.claude/commands/` 目录  
B) 每个开发者的 `~/.claude/commands/`  
C) 项目根目录的 CLAUDE.md  
D) `.claude/config.json` 的 commands 数组  

> **答案 A**

> - A 项目 `.claude/commands/` 受版本控制，克隆/拉取时自动可用
> - B ❌ `~/.claude/commands/` 是个人命令，不共享
> - C ❌ CLAUDE.md 是项目指令和上下文，不是命令定义
> - D ❌ 该配置机制**不存在**（典型的干扰项模式：不存在的功能）

### 样题 5：计划模式 vs 直接执行 〔领域 3 · 任务 3.4〕

> **场景**：将团队单体应用重构为微服务，涉及数十个文件更改和架构决策。

A) 进入计划模式，探索代码库和依赖后再更改  
B) 直接执行逐步更改  
C) 直接执行加全面前期指令  
D) 直接执行开始，遇到复杂性再切换  

> **答案 A**
>
> **解题思路**：三个关键词——"重构""数十个文件""架构决策"→ 大规模 + 不确定性高。
>
> - A 大规模更改 + 架构决策 = 计划模式（先探索再行动）
> - B ❌ 直接开干可能在发现关键依赖时代价高昂地返工
> - C ❌ 假设你在未探索代码时就知道正确结构
> - D ❌ 题目明确说了"数十个文件"，复杂性已经很明确，不是"可能浮现"
>
> **核心原则**：大规模 + 架构决策 → 计划模式；小规模 + 明确修改 → 直接执行
