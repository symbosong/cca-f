Claude 认证架构师 – 基础认证（CCA-F）是 Anthropic 官方推出的认证考试，验证从业者在使用 Claude 实现实际解决方案时**能够就权衡做出明智决策**的能力。

> **这是考试权重最高的领域。** 核心考察使用 Claude Agent SDK 构建和编排智能体系统的能力。
>
> **补充学习资源**：
> - [Agent SDK 文档](https://platform.claude.com/docs/en/agent-sdk/overview)
> - [Introduction to subagents 课程](https://anthropic.skilljar.com/introduction-to-subagents)（20min）
> - [Introduction to agent skills 课程](https://anthropic.skilljar.com/introduction-to-agent-skills)（30min）
> - 📄 进阶模块参考：`09_Agent_SDK深度指南.md`

### 1.1 智能体循环（Agent Loop）

**考试要求**：设计和实现用于自主任务执行的智能体循环

**核心知识**：
- 智能体循环生命周期：向 Claude 发送请求 → 检查 `stop_reason` → 执行工具 → 返回结果 → 循环
- 模型驱动的决策（Claude 自主决定调用哪个工具）vs 预配置决策树
- 工具结果如何附加到对话历史中

**stop_reason 处理（高频考点）**：

| stop_reason 值 | 含义 | 处理方式 |
|:---|:---|:---|
| `end_turn` | 模型正常完成 | 循环终止，返回结果 |
| `tool_use` | 模型请求调用工具 | 执行工具，结果送回，继续循环 |
| `max_tokens` | 达到输出 Token 限制 | 循环终止，检查是否需要续写 |

**正确的控制流**：

```python
while True:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        messages=messages, tools=tools
    )
    if response.stop_reason == "end_turn":
        break  # ✅ 模型完成，退出循环
    elif response.stop_reason == "tool_use":
        # 执行工具 → 结果加入上下文 → 继续
        tool_results = execute_tools(response.content)
        messages.append({"role": "assistant", "content": response.content})
        messages.append({"role": "user", "content": tool_results})
```

**必须避免的反模式**：
1. ❌ 解析自然语言信号确定循环终止（如检查"我完成了"）
2. ❌ 任意迭代上限作为主要停止机制（如最多循环 10 次）
3. ❌ 检查助手文本内容作为完成指示器

### 1.2 协调器-子智能体编排

**考试要求**：使用协调器-子智能体模式编排多智能体系统

**核心架构（中心辐射模式）**：

```
          ┌──────────────┐
          │   协调器       │
          │ (Coordinator) │
          └──────┬───────┘
         ┌───────┼───────┐
         ▼       ▼       ▼
    ┌────────┐┌────────┐┌────────┐
    │ 子智能体 ││ 子智能体 ││ 子智能体 │
    │  搜索   ││  分析   ││  综合   │
    └────────┘└────────┘└────────┘
```

**关键规则**：
- 协调器管理**所有**子智能体间通信、错误处理和信息路由
- 子智能体**隔离运行**——不自动继承协调器对话历史
- 上下文必须**显式传递**——子智能体不共享内存

**常见考点 — 过窄的任务分解**：
> 场景：研究"AI 对创意产业的影响"，但最终报告仅涵盖视觉艺术。
> 根因：协调器将"创意产业"仅分解为视觉艺术子类（数字艺术、平面设计、摄影），遗漏了音乐、写作、电影。

### 1.3 子智能体配置与上下文传递

**考试要求**：配置子智能体调用、上下文传递和生成

- **Task 工具**：生成子智能体的机制，`allowedTools` 必须包含 `"Task"` 才能嵌套子智能体
- 子智能体上下文必须显式提供——不自动继承父上下文
- 使用结构化数据格式分离**内容与元数据**（源 URL、文档名、页码）以保留归属
- 在单个协调器响应中发出多个 Task 工具调用 → **并行**子智能体生成

### 1.4 强制模式与交接协议

**核心考点 — 程序化强制 vs 提示引导**：

| 方式 | 可靠性 | 适用场景 |
|:---|:---|:---|
| **程序化强制**（钩子、前置条件门） | ✅ 确定性保证 | 业务规则必须遵守时 |
| **提示引导**（系统提示指令） | ⚠️ 概率性合规 | 软性偏好和风格指导 |

**考试场景**：智能体跳过 `get_customer` 直接调用退款 → **正确答案**：程序化前置条件（阻止退款直到验证完成），不是增强提示或添加示例。

### 1.5 钩子（Hooks）

**两种钩子层面（必须区分）**：

| 维度 | Claude Code Hooks | Agent SDK Hooks |
|:---|:---|:---|
| **运行环境** | Claude Code CLI | 自定义 Agent 应用 |
| **配置方式** | JSON 文件（`settings.json`） | Python/TypeScript 代码 |
| **处理器类型** | command/http/prompt/agent | 代码函数 |

**决策原则**：业务规则需要保证合规时 → **用钩子**（确定性）；软性偏好 → **用提示**（概率性）

### 1.6 任务分解策略

| 模式 | 适用场景 |
|:---|:---|
| **提示链（固定顺序流水线）** | 可预测的审查/分析：按文件分析 → 跨文件集成遍历 |
| **动态自适应分解** | 开放式调查：先映射结构 → 识别重点 → 自适应计划 |

### 1.7 会话状态管理

| 命令/功能 | 用途 |
|:---|:---|
| `--resume <session-name>` | 恢复命名会话 |
| `fork_session` | 从共享基线创建独立分支 |
| `/compact` | 压缩上下文，减少 Token 使用 |
| `/memory` | 查看加载的记忆文件 |

## 领域 2：工具设计与 MCP 集成（18%）

> **补充学习资源**：
> - [Introduction to MCP 课程](https://anthropic.skilljar.com/introduction-to-model-context-protocol)（1h）
> - [MCP: Advanced Topics 课程](https://anthropic.skilljar.com/model-context-protocol-advanced-topics)（1.1h）
> - 📄 基础课程参考：`03_工具使用与高级特性.md`、`04_MCP模型上下文协议.md`

### 2.1 工具接口设计

**核心原则**：工具描述是 LLM 工具选择的**主要机制**

```json
// ❌ 过于简约 → 不可靠的工具选择
{"name": "get_customer", "description": "检索客户信息"}

// ✅ 详细描述 → 可靠的工具选择
{"name": "get_customer", "description": "根据客户 ID、邮箱或电话号码检索客户账户信息。返回姓名、联系方式、账户状态。不返回订单数据——订单查询请使用 lookup_order。"}
```

### 2.2 MCP 错误响应

**错误分类**：

| 错误类型 | isRetryable | 处理方式 |
|:---|:---|:---|
| 瞬态错误（超时、网络） | `true` | 子智能体本地重试 |
| 验证错误（参数无效） | `false` | 返回修正建议 |
| 业务规则违反 | `false` | 返回客户友好解释 |

**关键区分**：访问失败（"数据库不可达"→应重试）≠ 有效空结果（"该客户无订单"→正常继续）

### 2.3 工具分配与 tool_choice

- 过多工具（18个）→ 选择可靠性差；4-5 个 → 可靠性高
- 限制子智能体工具集到其角色相关范围（**最小权限原则**）

| tool_choice 值 | 行为 | 适用场景 |
|:---|:---|:---|
| `"auto"` | 模型自动决定 | 默认行为 |
| `"any"` | 强制使用工具 | 保证结构化输出 |
| `{"type": "tool", "name": "xxx"}` | 强制用特定工具 | 确保提取在富化前运行 |

### 2.4 MCP 服务器集成

| 范围 | 配置文件 | 特点 |
|:---|:---|:---|
| **项目级** | `.mcp.json` | 版本控制共享，团队可用 |
| **用户级** | `~/.claude.json` | 个人使用，跨项目 |

- 环境变量展开：`${GITHUB_TOKEN}`
- 所有已配置 MCP 服务器的工具**同时可用**
- MCP 资源暴露内容目录以减少探索性工具调用

### 2.5 内置工具

| 工具 | 用途 | 关键区分 |
|:---|:---|:---|
| **Grep** | 内容搜索 | 搜索代码**内容** |
| **Glob** | 文件路径匹配 | 查找匹配的**文件路径** |
| **Edit** | 有针对性修改 | 非唯一匹配失败时用 Read + Write |

## 领域 3：Claude Code 配置与工作流（20%）

> **补充学习资源**：
> - [Claude Code in Action 课程](https://anthropic.skilljar.com/claude-code-in-action)（1h）— **必学！**
> - 📄 进阶模块参考：`10_Claude_Code高级特性.md`

### 3.1 CLAUDE.md 三级配置体系（高频考点）

| 层级 | 位置 | 共享方式 | 用途 |
|:---|:---|:---|:---|
| **项目级** | `.claude/CLAUDE.md` | 版本控制（团队共享） | 项目架构、编码标准 |
| **用户级** | `~/.claude/CLAUDE.md` | 不共享（仅个人） | 个人偏好 |
| **组织级** | 系统级路径 | 管理员控制 | 公司安全策略 |

**常见考试场景**：新成员 Claude 不遵循编码标准 → 配置写在了用户级 → 移到项目级共享。

- `@import` 语法引用外部文件（最多 5 层递归）
- `.claude/rules/` 目录用 YAML frontmatter 的 `paths` 字段 + glob 模式

### 3.2 自定义命令与技能

| 类型 | 位置 | 共享方式 |
|:---|:---|:---|
| **项目命令** | `.claude/commands/` | 版本控制共享 |
| **个人命令** | `~/.claude/commands/` | 仅个人 |

**SKILL.md 配置**：
- `context: fork` — 在隔离子智能体上下文中运行
- `allowed-tools` — 限制技能可用的工具
- `argument-hint` — 提示所需参数

**技能 vs CLAUDE.md 的选择**：技能是**按需调用**的任务工作流；CLAUDE.md 是**始终加载**的通用标准。

### 3.3 路径特定规则

```markdown
---
paths:
  - "**/*.test.tsx"      # 所有测试文件
  - "terraform/**/*"      # 所有 Terraform 文件
---
```

**考试场景**：测试文件分散在代码库各处 → **正确答案**：`.claude/rules/` + glob 模式（不是每个子目录放 CLAUDE.md）

### 3.4 计划模式 vs 直接执行

| 使用计划模式 | 使用直接执行 |
|:---|:---|
| 大规模更改（45+ 文件）、架构决策、微服务重构 | 简单明确的修复、单文件修改 |

**考试场景**：单体应用重构为微服务 → **计划模式**

### 3.5 迭代细化技术

| 技巧 | 说明 |
|:---|:---|
| **具体示例** | 提供 2-3 个输入/输出示例传达预期转换 |
| **测试驱动** | 先编写测试，通过失败引导改进 |
| **面试模式** | 让 Claude 提问发掘未预见的考虑因素 |

### 3.6 CI/CD 集成

| CLI 标志 | 用途 |
|:---|:---|
| `-p` / `--print` | **非交互模式**（CI/CD 必须！） |
| `--output-format json` | 生成 JSON 格式输出 |
| `--json-schema` | 强制输出符合指定模式 |

**考试场景**：流水线脚本 `claude "分析 PR..."` 挂起 → **答案：加 `-p` 标志**

## 领域 4：提示工程与结构化输出（20%）

> **基础课程参考**：`02_提示工程与评估.md`、`03_工具使用与高级特性.md`
> **进阶模块参考**：`12_Prompt_Caching与Batch_API.md`

### 4.1 明确标准的提示设计

```
❌ "检查代码是否准确"
❌ "保守一些"
✅ "仅在行为矛盾时标记；跳过次要风格问题"
```

### 4.2 少样本提示

少样本示例是实现一致格式化输出**最有效**的技术——适用于歧义处理、格式一致性、减少幻觉。

### 4.3 结构化输出（tool_use + JSON 模式）

- `tool_use + strict: true` 是保证模式兼容输出的**最可靠方法**
- **可空字段** `"type": ["string", "null"]` — 防止捏造不存在的值
- **枚举 + "other"** — 涵盖已知类别 + 未知情况
- **"unclear" 枚举值** — 允许模型表达不确定性
- `strict: true` 消除语法错误，但**不防止语义错误**

### 4.4 验证-重试-反馈循环

- 重试时附带：原文档 + 失败提取 + **具体**验证错误
- 重试的局限：信息不存在于源文档时 → **不要重试**

### 4.5 批处理策略

| 参数 | 值 |
|:---|:---|
| **Message Batches API 成本节省** | 50%（可与 Prompt Caching 叠加） |
| **最长处理时间** | 24 小时 |
| **多轮工具调用** | ❌ 不支持 |

**考试场景**：阻塞合并前检查 + 隔夜技术债务报告 → 仅技术债务用批处理，合并检查保持实时。

### 4.6 多实例/多遍审查

- **自我审查的局限**：保留推理上下文，不太可能质疑自己
- **正确做法**：独立审查实例（不带生成器推理上下文）
- **多遍审查**：按文件本地分析 + 跨文件集成遍历

## 领域 5：上下文管理与可靠性（15%）

> **进阶模块参考**：`13_安全护栏与生产部署.md`

### 5.1 上下文管理

**三大风险**：

| 风险 | 缓解措施 |
|:---|:---|
| 渐进摘要失真 | 将关键事实提取到持久"案例事实"块 |
| 迷失在中间 | 关键发现放在开头 + 明确标题 |
| 工具结果膨胀 | 修剪为仅相关字段 |

### 5.2 升级与歧义解决

**适当升级触发器**：

| 触发器 | 处理方式 |
|:---|:---|
| 客户明确要求人工 | **立即升级** |
| 策略例外/策略空白 | 升级 |
| 无法进展 | 升级 |
| 客户表达不满 | 承认挫折 + 提供方案，仅在客户重申时升级 |

**不可靠的升级方式**：
- ❌ 基于情绪的升级 — 情绪与案例复杂性不相关
- ❌ 自我报告置信度 — LLM 的置信度校准很差

### 5.3 错误传播策略

- 返回包含失败类型、尝试内容、部分结果和替代方案的**结构化错误**
- 区分访问失败和有效空结果
- 子智能体**本地恢复**瞬态故障，仅传播无法解决的错误
- **反模式**：静默抑制错误、单一失败终止整个工作流

### 5.4 大型代码库上下文管理

- **子智能体委托**：隔离冗长探索，主智能体保持高级协调
- **草稿本文件**：跨上下文边界持久化关键发现
- **结构化状态导出**：用于崩溃恢复

### 5.5 人工审查与置信度校准

- 总体准确率可能掩盖特定文档类型/字段的性能不佳
- **分层随机抽样**测量错误率
- **字段级置信度**使用标记验证集校准阈值

### 5.6 信息来源与不确定性

- 子智能体输出**声明-来源映射**
- 冲突统计：使用源归属注释，**不要**任意选择
- 要求出版/收集日期防止误解

## 样题与解题策略

### 解题框架

| 类型 | 标志词 | 答题策略 |
|:---|:---|:---|
| **根因分析** | "最可能的根因" | 从证据出发，锁定根因 |
| **最佳方案** | "最有效"、"最可靠" | 排除过度工程化和不足的方案 |
| **配置选择** | "应该在哪里" | 匹配范围需求（项目/用户/组织） |
| **首要步骤** | "第一步" | 选低成本高杠杆的方案 |

### 排除干扰项的常见模式

| 干扰项模式 | 例子 |
|:---|:---|
| 过度工程化 | "部署独立分类器" |
| 不存在的功能 | `CLAUDE_HEADLESS=true`、`--batch` |
| 解决错误问题 | 情绪分析解决升级问题 |
| 概率性方案解决确定性需求 | 用提示保证退款前验证 |
| 归咎下游 | 子智能体正确但被分配了错误任务 |

### 核心判断原则

| 原则 | 说明 |
|------|------|
| **确定性优先** | 业务关键逻辑用钩子强制，不要靠提示 |
| **最小权限** | 每个子智能体只给它需要的工具 |
| **先描述后代码** | 工具选错了？先改描述（低成本高杠杆） |
| **独立审查优于自审** | 自我审查保留了推理上下文 |
| **结构化一切** | 错误、交接、输出、来源——全部结构化 |

### 样题精选

#### 样题 1：程序化强制 vs 提示引导

> **场景**：12% 的案例中智能体跳过客户验证直接退款。什么变更最有效？
>
> A) **添加程序化前置条件阻止退款直到验证完成** ✅  
> B) 增强系统提示声明验证是强制性的  
> C) 添加少样本示例展示总是先验证  
> D) 实现路由分类器仅启用适合的工具子集

**答案 A**：确定性需求 → 程序化强制。B、C 是概率性合规，有财务风险。D 解决的是工具可用性不是排序。

#### 样题 2：工具描述优化

> **场景**：用户问订单时智能体频繁调用 get_customer 而非 lookup_order。两个工具描述都很简约。改善工具选择的第一步？
>
> A) 5-8 个少样本示例  
> B) **扩展每个工具描述包含输入格式、示例和边界** ✅  
> C) 基于关键词的路由层  
> D) 合并为单个工具

**答案 B**：工具描述是 LLM 选择工具的主要机制。作为"第一步"，改描述是低成本高杠杆。

#### 样题 3：升级校准

> **场景**：智能体升级简单案例却自主处理需要策略例外的复杂情况。如何改进？
>
> A) **添加带少样本示例的明确升级标准** ✅  
> B) 自我报告置信度分数  
> C) 部署独立分类器  
> D) 情绪分析检测挫折度

**答案 A**：根因是不清晰的决策边界。B 校准差，C 过度工程化，D 解决了错误问题。

#### 样题 4：项目命令位置

> **场景**：创建 /review 命令让团队克隆后自动可用。文件放哪里？
>
> A) **项目仓库的 `.claude/commands/` 目录** ✅  
> B) 每个开发者的 `~/.claude/commands/`  
> C) 项目根目录的 CLAUDE.md  
> D) `.claude/config.json` 的 commands 数组

**答案 A**：项目范围命令受版本控制，克隆/拉取时自动可用。B 是个人命令，D 不存在。

#### 样题 5：计划模式 vs 直接执行

> **场景**：将团队单体应用重构为微服务，涉及数十个文件更改和架构决策。
>
> A) **进入计划模式探索代码库和依赖后再更改** ✅  
> B) 直接执行逐步更改  
> C) 直接执行加全面前期指令  
> D) 直接执行开始遇到复杂性再切换

**答案 A**：大规模 + 架构决策 → 计划模式。
