# Claude 认证架构师 – 基础认证考试指南

## 简介

Claude 认证架构师 – 基础认证验证了从业者在使用 Claude 实现实际解决方案时能够就权衡做出明智决策的能力。本考试测试关于 Claude Code、Claude Agent SDK、Claude API 和模型上下文协议(MCP)的基础知识——这些是使用 Claude 构建生产级应用的核心技术。

本考试的问题基于实际客户用例的真实场景，包括为客户支持构建智能体系统、设计多智能体研究流程、将 Claude Code 集成到 CI/CD 工作流中、构建开发者生产力工具，以及从非结构化文档中提取结构化数据。考生必须不仅展示概念知识，还要展示在生产环境中关于架构、配置和权衡的实践判断能力。

本指南描述了考试内容，列出了测试的领域和任务陈述，提供了样题，并推荐了准备策略。将其与实际经验相结合可以有效地准备考试。

## 目标候选人描述

本认证的理想候选人是设计和实施 Claude 生产应用的解决方案架构师。该候选人具有以下实践经验：

- 使用 Claude Agent SDK 构建智能体应用，包括多智能体编排、子智能体委托、工具集成和生命周期钩子
- 使用 CLAUDE.md 文件、Agent Skills、MCP 服务器集成和计划模式为团队工作流配置和自定义 Claude Code
- 为后端系统集成设计模型上下文协议(MCP)工具和资源接口
- 设计能够生成可靠结构化输出的提示，利用 JSON 模式、少样本示例和提取模式
- 在长文档、多轮对话和多智能体交接中有效管理上下文窗口
- 将 Claude 集成到 CI/CD 流水线中，用于自动代码审查、测试生成和拉取请求反馈
- 做出合理的升级和可靠性决策，包括错误处理、人工参与工作流和自评估模式

该候选人通常拥有 6 个月以上使用 Claude API、Agent SDK、Claude Code 和 MCP 构建的实际经验，了解大型语言模型在生产环境中的能力和局限性。

## 考试内容

### 回答类型

所有考试问题均为单项选择题（multiple choice）。每道题有一个正确答案和三个错误选项（干扰项）。

选择最能完成陈述或回答问题的单个选项。干扰项是知识或经验不完整的考生可能会选择的选项。

未回答的问题记为错误；猜测没有扣分惩罚。

### 考试结果

考试采用及格/不及格制。考试根据由主题专家建立的最低标准进行评分。

你的成绩以 100-1,000 的缩放分数报告。**最低通过分数为 720 分**。

缩放评分模型有助于在可能具有略微不同难度级别的多个考试表单之间等效分数。

### 内容大纲

本考试指南包含考试的权重、内容领域和任务陈述。

考试具有以下内容领域和权重：

- **领域 1：智能体架构与编排（占评分内容的 27%）**
- **领域 2：工具设计与 MCP 集成（占评分内容的 18%）**
- **领域 3：Claude Code 配置与工作流（占评分内容的 20%）**
- **领域 4：提示工程与结构化输出（占评分内容的 20%）**
- **领域 5：上下文管理与可靠性（占评分内容的 15%）**

### 考试场景

考试使用基于场景的问题。每个场景呈现一个真实的生产环境，用于构建一组问题。考试期间，将从以下 6 个完整场景中随机抽取 4 个场景。

**场景 1：客户支持解决智能体**
你正在使用 Claude Agent SDK 构建客户支持解决智能体。该智能体处理高歧义性请求，如退货、账单争议和账户问题。它通过自定义 MCP 工具（get_customer、lookup_order、process_refund、escalate_to_human）访问后端系统。目标是 80%+ 首次联系解决率，同时知道何时升级。
主要领域：智能体架构与编排、工具设计与 MCP 集成、上下文管理与可靠性

**场景 2：使用 Claude Code 生成代码**
你正在使用 Claude Code 加速软件开发。团队将其用于代码生成、重构、调试和文档。需要集成到开发工作流中，使用自定义斜杠命令、CLAUDE.md 配置，并了解何时使用计划模式与直接执行。
主要领域：Claude Code 配置与工作流、上下文管理与可靠性

**场景 3：多智能体研究系统**
你正在使用 Claude Agent SDK 构建多智能体研究系统。协调器智能体委托给专门的子智能体：搜索网络、分析文档、综合发现和生成报告。系统研究主题并生成全面的有引用报告。
主要领域：智能体架构与编排、工具设计与 MCP 集成、上下文管理与可靠性

**场景 4：使用 Claude 的开发者生产力工具**
你正在使用 Claude Agent SDK 构建开发者生产力工具。智能体帮助工程师探索不熟悉的代码库、理解遗留系统、生成样板代码和自动化重复任务。使用内置工具（Read、Write、Bash、Grep、Glob）并与 MCP 服务器集成。
主要领域：工具设计与 MCP 集成、Claude Code 配置与工作流、智能体架构与编排

**场景 5：用于持续集成的 Claude Code**
你正在将 Claude Code 集成到 CI/CD 流水线中。系统运行自动化代码审查、生成测试用例并提供拉取请求反馈。需要设计提供可操作反馈并最小化误报的提示。
主要领域：Claude Code 配置与工作流、提示工程与结构化输出

**场景 6：结构化数据提取**
你正在使用 Claude 构建结构化数据提取系统。系统从非结构化文档提取信息，使用 JSON 模式验证输出并保持高准确性。必须优雅处理边缘情况并与下游系统集成。
主要领域：提示工程与结构化输出、上下文管理与可靠性

## 领域 1：智能体架构与编排

### 任务陈述 1.1：设计和实现用于自主任务执行的智能体循环

**知识点：**
- 智能体循环生命周期：向 Claude 发送请求、检查 stop_reason（"tool_use" vs "end_turn"）、执行请求的工具、返回结果进入下一次迭代
- 工具结果如何附加到对话历史中，以便模型推理下一步操作
- 模型驱动的决策（Claude 根据上下文推理调用哪个工具）与预配置决策树或工具序列之间的区别

**技能要求：**
- 实现智能体循环控制流，当 stop_reason 为 "tool_use" 时继续，为 "end_turn" 时终止
- 在迭代之间将工具结果添加到对话上下文中
- 避免反模式：解析自然语言信号确定循环终止、任意迭代上限作为主要停止机制、检查助手文本内容作为完成指示器

### 任务陈述 1.2：使用协调器-子智能体模式编排多智能体系统

**知识点：**
- 中心辐射架构，协调器管理所有子智能体间通信、错误处理和信息路由
- 子智能体以隔离上下文运行——不自动继承协调器对话历史
- 协调器在任务分解、委托、结果聚合和根据查询复杂性选择子智能体中的角色
- 过于狭窄的任务分解风险，导致广泛研究主题的覆盖不完整

**技能要求：**
- 设计动态选择子智能体的协调器，而非总是通过完整流水线路由
- 在子智能体间划分研究范围以最小化重复
- 实现迭代细化循环：评估综合输出的差距，使用针对性查询重新委托，直到覆盖充分
- 通过协调器路由所有子智能体通信以实现可观测性和受控信息流

### 任务陈述 1.3：配置子智能体调用、上下文传递和生成

**知识点：**
- Task 工具作为生成子智能体的机制，allowedTools 必须包含 "Task"
- 子智能体上下文必须显式提供——不自动继承父上下文或共享内存
- AgentDefinition 配置：描述、系统提示和工具限制
- 基于分支的会话管理，从共享分析基线探索不同方法

**技能要求：**
- 在子智能体提示中直接包含先前智能体的完整发现
- 使用结构化数据格式分离内容与元数据（源 URL、文档名、页码）以保留归属
- 在单个协调器响应中发出多个 Task 工具调用实现并行子智能体生成
- 设计指定研究目标和质量标准（而非逐步程序指令）的协调器提示

### 任务陈述 1.4：实现带有强制和交接模式的多步工作流

**知识点：**
- 程序化强制（钩子、前置条件门）与基于提示引导之间的区别
- 确定性合规场景中提示指令有非零失败率
- 中途升级的结构化交接协议

**技能要求：**
- 实现程序化前置条件阻止下游工具调用（如 get_customer 验证前阻止 process_refund）
- 将多关注点请求分解为独立项目并行调查后综合
- 编译结构化交接摘要（客户 ID、根因、退款金额、建议操作）

### 任务陈述 1.5：应用 Agent SDK 钩子进行工具调用拦截和数据规范化

**知识点：**
- PostToolUse 钩子模式，在模型处理前拦截转换工具结果
- 拦截外出工具调用以强制合规规则的钩子模式
- 钩子（确定性保证）vs 提示指令（概率性合规）

**技能要求：**
- 实现 PostToolUse 钩子规范化异构数据格式（Unix 时间戳、ISO 8601、数字状态码）
- 实现工具调用拦截钩子阻止违规操作并重定向到替代工作流
- 业务规则需要保证合规时选择钩子而非提示

### 任务陈述 1.6：为复杂工作流设计任务分解策略

**知识点：**
- 固定顺序流水线（提示链）vs 动态自适应分解
- 提示链模式：将审查分解为顺序步骤
- 自适应调查计划的价值

**技能要求：**
- 根据工作流选择分解模式：提示链用于可预测审查，动态分解用于开放式调查
- 拆分大型代码审查为按文件本地分析加跨文件集成遍历
- 分解开放式任务：先映射结构、识别高影响区域，然后创建自适应优先计划

### 任务陈述 1.7：管理会话状态、恢复和分支

**知识点：**
- --resume <session-name> 命名会话恢复
- fork_session 从共享基线创建独立分支
- 恢复会话时告知文件更改的重要性
- 结构化摘要开始新会话 vs 在过时工具结果中恢复

**技能要求：**
- 使用 --resume 跨工作会话继续命名调查会话
- 使用 fork_session 创建并行探索分支
- 在恢复和注入摘要新会话之间选择
- 告知恢复会话具体文件更改以进行针对性重新分析

## 领域 2：工具设计与 MCP 集成

### 任务陈述 2.1：设计具有清晰描述和边界的有效工具接口

**知识点：**
- 工具描述是 LLM 工具选择的主要机制；最小化描述导致不可靠选择
- 包含输入格式、示例查询、边缘情况和边界解释的重要性
- 模糊/重叠工具描述导致错误路由
- 系统提示关键词敏感指令对工具选择的影响

**技能要求：**
- 编写清楚区分用途、输入、输出和使用场景的工具描述
- 重命名工具消除功能重叠
- 拆分通用工具为目的特定工具并定义输入/输出契约
- 审查系统提示中可能覆盖工具描述的关键词敏感指令

### 任务陈述 2.2：为 MCP 工具实现结构化错误响应

**知识点：**
- MCP isError 标志模式
- 瞬态/验证/业务/权限错误的区别
- 统一错误响应阻止适当恢复决策
- 可重试 vs 不可重试错误

**技能要求：**
- 返回包含 errorCategory、isRetryable 和人类可读描述的结构化错误元数据
- 为业务规则违反包含 retriable: false 和客户友好解释
- 子智能体本地恢复瞬态故障，仅传播无法解决的错误
- 区分访问故障和有效空结果

### 任务陈述 2.3：在智能体间适当分配工具并配置工具选择

**知识点：**
- 过多工具（18 个 vs 4-5 个）降低选择可靠性
- 专业外工具的误用倾向
- 范围化工具访问原则
- tool_choice 选项："auto"、"any"、强制选择

**技能要求：**
- 限制子智能体工具集到其角色相关范围
- 用受限替代方案替换通用工具
- 为高频需求提供范围化跨角色工具
- 使用 tool_choice 强制选择和 "any" 保证工具调用

### 任务陈述 2.4：将 MCP 服务器集成到 Claude Code 和智能体工作流

**知识点：**
- MCP 服务器范围：项目级 .mcp.json vs 用户级 ~/.claude.json
- 环境变量展开（${GITHUB_TOKEN}）
- 所有已配置 MCP 服务器工具同时可用
- MCP 资源暴露内容目录以减少探索性工具调用

**技能要求：**
- 配置项目范围和用户范围 MCP 服务器
- 增强 MCP 工具描述防止偏好内置工具
- 选择社区 MCP 服务器 vs 自定义实现
- 暴露内容目录为 MCP 资源

### 任务陈述 2.5：有效选择和应用内置工具

**知识点：**
- Grep 用于内容搜索，Glob 用于文件路径模式匹配
- Read/Write 用于完整文件操作，Edit 用于有针对性修改
- Edit 非唯一文本匹配失败时的 Read + Write 后备

**技能要求：**
- Grep 搜索代码内容，Glob 查找匹配文件
- Edit 失败时使用 Read + Write
- 增量构建代码库理解：Grep 找入口点 → Read 跟踪导入
- 跨包装模块追踪函数使用

## 领域 3：Claude Code 配置与工作流

### 任务陈述 3.1：使用适当的层级、范围和模块化组织配置 CLAUDE.md 文件

**知识点：**
- CLAUDE.md 配置层级：用户级（~/.claude/CLAUDE.md）、项目级（.claude/CLAUDE.md 或根目录 CLAUDE.md）、目录级（子目录 CLAUDE.md）
- 用户级设置仅适用于该用户，不通过版本控制共享
- @import 语法引用外部文件保持模块化
- .claude/rules/ 目录组织主题特定规则文件

**技能要求：**
- 诊断配置层级问题（如用户级 vs 项目级配置导致新成员缺少指令）
- 使用 @import 有选择地包含标准文件
- 将大型 CLAUDE.md 拆分为 .claude/rules/ 中的主题文件（testing.md、api-conventions.md 等）
- 使用 /memory 命令验证加载的记忆文件并诊断不一致行为

### 任务陈述 3.2：创建和配置自定义斜杠命令和技能

**知识点：**
- 项目范围命令 .claude/commands/（版本控制共享）vs 用户范围 ~/.claude/commands/（个人）
- .claude/skills/ 中的 SKILL.md 支持 frontmatter：context: fork、allowed-tools、argument-hint
- context: fork 在隔离子智能体上下文中运行技能
- 个人技能自定义：~/.claude/skills/ 中不同名称的变体

**技能要求：**
- 创建项目范围斜杠命令实现团队可用
- 使用 context: fork 隔离产生冗长输出的技能
- 配置 allowed-tools 限制技能执行期间的工具访问
- 使用 argument-hint 提示所需参数
- 技能（按需任务工作流）vs CLAUDE.md（始终加载通用标准）的选择

### 任务陈述 3.3：为条件约定加载应用路径特定规则

**知识点：**
- .claude/rules/ 文件的 YAML frontmatter paths 字段含 glob 模式
- 路径范围规则仅在编辑匹配文件时加载
- glob 模式规则相对于目录级 CLAUDE.md 的优势

**技能要求：**
- 创建带路径范围的规则文件（如 paths: ["terraform/**/*"]）
- 使用 glob 模式按类型应用约定（如 **/*.test.tsx）
- 分布式文件时选择路径特定规则而非子目录 CLAUDE.md

### 任务陈述 3.4：确定何时使用计划模式与直接执行

**知识点：**
- 计划模式用于大规模更改、多种方法、架构决策、多文件修改
- 直接执行用于简单、范围明确的更改
- 计划模式允许安全探索和设计，防止昂贵返工
- Explore 子智能体隔离冗长发现输出并返回摘要

**技能要求：**
- 架构影响任务选择计划模式（微服务重构、45+ 文件库迁移等）
- 明确范围的已理解更改选择直接执行
- Explore 子智能体防止上下文窗口耗尽
- 计划模式调查 + 直接执行实现的组合

### 任务陈述 3.5：应用迭代细化技术进行渐进式改进

**知识点：**
- 具体输入/输出示例是传达预期转换的最有效方式
- 测试驱动迭代：先编写测试，通过共享失败引导改进
- 面试模式：让 Claude 提问发掘未预见的考虑因素
- 单消息所有问题（交互问题）vs 顺序修复（独立问题）

**技能要求：**
- 提供 2-3 个具体输入/输出示例澄清转换需求
- 先编写测试套件再实现，通过测试失败迭代
- 面试模式发掘设计考虑（缓存失效、故障模式等）
- 提供具体测试用例（含示例输入和预期输出）修复边缘情况处理（如迁移脚本中的 null 值）
- 交互修复在单消息中处理，独立问题顺序迭代

### 任务陈述 3.6：将 Claude Code 集成到 CI/CD 流水线

**知识点：**
- -p / --print 标志非交互模式
- --output-format json 和 --json-schema 强制结构化输出
- CLAUDE.md 提供 CI 上下文（测试标准、fixture 约定、审查标准）
- 会话上下文隔离：同一会话审查自己更改效果差

**技能要求：**
- CI 中使用 -p 标志防止交互挂起
- --output-format json + --json-schema 生成机器可解析的 PR 内联评论
- 重新审查时包含先前发现，仅报告新/未解决问题
- 提供现有测试文件避免重复测试建议
- CLAUDE.md 中记录测试标准和 fixture

## 领域 4：提示工程与结构化输出

### 任务陈述 4.1：设计具有明确标准的提示以提高精度并减少误报

**知识点：**
- 明确标准比模糊指令重要（"仅在行为矛盾时标记" vs "检查是否准确"）
- "保守一些"等通用指令无法比具体分类标准更好地提高精度
- 高误报率破坏开发者信任

**技能要求：**
- 编写定义报告问题（bug、安全）vs 跳过问题（次要风格）的具体审查标准
- 暂时禁用高误报类别恢复信任
- 为每个严重级别定义带代码示例的明确标准

### 任务陈述 4.2：应用少样本提示以提高输出一致性和质量

**知识点：**
- 少样本示例是实现一致格式化可操作输出的最有效技术
- 展示歧义案例处理的角色
- 泛化判断到新模式的能力
- 减少提取任务中幻觉的有效性

**技能要求：**
- 创建 2-4 个针对歧义场景的少样本示例，展示选择推理
- 包含展示期望输出格式的示例
- 区分可接受模式与真正问题的示例
- 展示不同文档结构正确处理的示例
- 添加展示从不同格式文档正确提取的少样本示例，以解决必填字段的空/null 提取问题

### 任务陈述 4.3：使用工具使用和 JSON 模式强制结构化输出

**知识点：**
- tool_use + JSON 模式是保证模式兼容输出的最可靠方法
- tool_choice："auto" vs "any" vs 强制选择
- 严格 JSON 模式消除语法错误但不防止语义错误
- 模式设计：必需/可选字段、枚举 + "other" 模式

**技能要求：**
- 定义 JSON 模式提取工具并从 tool_use 响应提取数据
- tool_choice: "any" 保证结构化输出（文档类型未知时）
- 强制特定工具确保提取在富化前运行
- 可选（可空）字段防止捏造值
- "unclear" 枚举值和 "other" + 详情字段
- 在严格输出模式旁提示中包含格式规范化规则，以处理不一致的源格式

### 任务陈述 4.4：实现验证、重试和反馈循环

**知识点：**
- 带错误反馈的重试：附加特定验证错误引导修正
- 重试局限：信息不存在于源文档时无效
- 反馈循环设计：detected_pattern 字段跟踪触发发现
- 语义验证错误 vs 模式语法错误

**技能要求：**
- 后续请求包含原文档、失败提取和特定验证错误
- 识别重试何时无效 vs 何时成功
- detected_pattern 字段启用误报模式分析
- 自我修正验证：calculated_total vs stated_total、conflict_detected 布尔值

### 任务陈述 4.5：设计高效的批处理策略

**知识点：**
- Message Batches API：50% 成本节省、最长 24 小时、无保证延迟 SLA
- 适用非阻塞延迟容忍工作负载，不适用阻塞工作流
- 不支持单请求内多轮工具调用
- custom_id 关联请求/响应

**技能要求：**
- 同步 API 用于阻塞检查，批处理 API 用于隔夜/每周分析
- 根据 SLA 计算批处理提交频率
- 通过 custom_id 识别失败文档并重新提交
- 批量处理前先在样本集上细化提示

### 任务陈述 4.6：设计多实例和多遍审查架构

**知识点：**
- 自我审查局限：保留推理上下文，不太可能质疑自己
- 独立审查实例比自我审查更有效
- 多遍审查：按文件本地分析 + 跨文件集成遍历

**技能要求：**
- 使用独立 Claude 实例审查（不带生成器推理上下文）
- 拆分为按文件遍历 + 集成遍历
- 验证遍历中自报告置信度启用校准审查路由

### 任务陈述 5.4：在大型代码库探索中有效管理上下文

**技能要求：**

**知识点：**

## 样题

以下样题说明了考试的格式和难度级别，从练习测试中抽取并包括解释。

### 场景：客户支持解决智能体

**问题 1：** 生产数据显示 12% 的案例中智能体跳过 get_customer 而仅用客户声称的名字调用 lookup_order，导致账户识别错误和不正确退款。什么变更最有效？

A) 添加程序化前置条件阻止 lookup_order 和 process_refund 直到 get_customer 返回验证的客户 ID
B) 增强系统提示声明 get_customer 验证是强制性的
C) 添加少样本示例展示总是先调用 get_customer
D) 实现路由分类器仅启用适合请求类型的工具子集

**正确答案：A** — 当关键业务逻辑（如处理退款前验证客户身份）需要特定工具序列时，程序化强制提供确定性保证，而基于提示的方法无法做到。选项 B 和 C 依赖概率性 LLM 合规，当错误具有财务后果时这是不够的。选项 D 解决的是工具可用性而非工具排序，不是实际问题。

**问题 2：** 生产日志显示用户问订单时（如"查一下我的订单 #12345"）智能体频繁调用 get_customer 而非 lookup_order。两个工具描述都很简约（"检索客户信息"/"检索订单详情"）且接受相似的标识符格式。改善工具选择可靠性最有效的第一步？

A) 5-8 个少样本示例展示正确工具选择
B) 扩展每个工具描述包含输入格式、示例查询、边缘情况和边界
C) 实现基于关键词的路由层预选工具
D) 合并为单个 lookup_entity 工具

**正确答案：B** — 工具描述是 LLM 工具选择的主要机制。当描述很简约时，模型缺乏区分相似工具的上下文。选项 B 用低成本、高杠杆的修复直接解决根因。少样本示例（A）在不修复根本问题的情况下增加令牌开销。路由层（C）过度工程化，绕过了 LLM 的自然语言理解能力。合并工具（D）是一个有效的架构选择，但作为"第一步"而言需要的工作量过多，而当前问题只是描述不充分。

**问题 3：** 智能体 55% 首次联系解决率（目标 80%+），日志显示它升级简单案例（有照片证据的标准损坏替换）却自主处理需要策略例外的复杂情况。如何改进升级校准？

A) 添加带少样本示例的明确升级标准
B) 自我报告置信度分数路由到人工
C) 部署独立分类器预测需要升级的请求
D) 情绪分析检测挫折程度自动升级

**正确答案：A** — 添加带少样本示例的明确升级标准直接解决根因：不清晰的决策边界。这是在增加基础设施之前的适当首次响应。选项 B 失败是因为 LLM 自我报告的置信度校准很差——智能体在困难案例上已经不正确地自信。选项 C 过度工程化，当提示优化还没尝试时就需要标注数据和 ML 基础设施。选项 D 完全解决的是不同的问题——情绪与案例复杂性不相关，而复杂性才是实际问题。

### 场景：使用 Claude Code 生成代码

**问题 4：** 你想创建一个自定义 /review 斜杠命令来运行团队的标准代码审查清单。该命令应该在每个开发者克隆或拉取仓库时自动可用。应在哪里创建该命令文件？

A) 项目仓库的 .claude/commands/ 目录
B) 每个开发者的 ~/.claude/commands/
C) 项目根目录的 CLAUDE.md 文件
D) .claude/config.json 的 commands 数组

**正确答案：A** — 项目范围自定义斜杠命令应存储在仓库内的 .claude/commands/ 目录中。这些命令受版本控制，开发者克隆或拉取时自动可用。选项 B（~/.claude/commands/）是个人命令，不通过版本控制共享。选项 C（CLAUDE.md）是项目指令和上下文，不是命令定义。选项 D 描述了 Claude Code 中不存在的配置机制。

**问题 5：** 你被安排将团队的单体应用重构为微服务。这将涉及数十个文件的更改，需要做出关于服务边界和模块依赖的决策。应采取哪种方法？

A) 进入计划模式探索代码库、理解依赖后再更改
B) 直接执行逐步更改让实现揭示服务边界
C) 直接执行加全面前期指令
D) 直接执行开始，遇到复杂性再切换计划模式

**正确答案：A** — 计划模式专为涉及大规模更改、多种有效方法和架构决策的复杂任务设计——这正是单体到微服务重构所需要的。它允许安全地探索代码库和设计后再做更改。选项 B 冒着依赖在后期发现时代价高昂返工的风险。选项 C 假设你在未探索代码的情况下已经知道正确结构。选项 D 忽略了复杂性已在需求中明确声明，而不是可能在后面浮现的东西。

**问题 6：** 你的代码库有不同区域使用不同编码约定：React 组件使用带 hooks 的函数式风格，API 处理器使用 async/await 加特定错误处理，数据库模型遵循仓储模式。测试文件分布在代码库各处紧挨着被测代码（如 Button.test.tsx 在 Button.tsx 旁边），你希望所有测试无论位置都遵循相同约定。确保 Claude 生成代码时自动应用正确约定的最可维护方式？

A) .claude/rules/ 中用 YAML frontmatter glob 模式
B) 根 CLAUDE.md 中合并所有约定依赖推断
C) .claude/skills/ 中为每种代码类型创建技能
D) 每个子目录放一个 CLAUDE.md

**正确答案：A** — 选项 A 正确，因为 .claude/rules/ 的 glob 模式（如 **/*.test.tsx）允许根据文件路径自动应用约定，不受目录位置限制——这对分布在整个代码库中的测试文件至关重要。选项 B 依赖推断而非显式匹配，不可靠。选项 C 需要手动调用技能或依赖 Claude 选择加载它们，与基于文件路径的确定性"自动"应用需求矛盾。选项 D 无法轻松处理分散在多个目录中的文件，因为 CLAUDE.md 文件是目录绑定的。

### 场景：多智能体研究系统

**问题 7：** 运行系统研究"AI 对创意产业的影响"后，你观察到每个子智能体都成功完成：网络搜索智能体找到相关文章，文档分析智能体正确总结论文，综合智能体产生连贯输出。然而最终报告仅涵盖视觉艺术，完全遗漏了音乐、写作和电影制作。检查协调器日志时，你看到它将主题分解为三个子任务："AI 在数字艺术创作中"、"AI 在平面设计中"和"AI 在摄影中"。最可能的根因？

A) 综合智能体缺乏识别覆盖差距的指令
B) 协调器任务分解过于狭窄
C) 网络搜索查询不够全面
D) 文档分析过滤了非视觉来源

**正确答案：B** — 协调器日志直接揭示了根因：它将"创意产业"仅分解为视觉艺术子任务（数字艺术、平面设计、摄影），完全遗漏了音乐、写作和电影。子智能体在分配范围内正确执行了任务——问题在于它们被分配了什么。选项 A、C 和 D 错误地将责任归咎于在其分配范围内正确工作的下游智能体。

**问题 8：** 网络搜索子智能体超时。如何设计失败信息流回协调器？

A) 返回结构化错误上下文（失败类型、查询、部分结果、替代方案）
B) 指数退避自动重试后返回通用"搜索不可用"
C) 捕获超时返回空结果标记为成功
D) 传播超时异常终止整个工作流

**正确答案：A** — 结构化错误上下文为协调器提供做出智能恢复决策所需的信息——是否用修改后的查询重试、尝试替代方法，或使用部分结果继续。选项 B 的通用状态隐藏了协调器的有价值上下文，阻止了知情决策。选项 C 通过将失败标记为成功来抑制错误，阻止任何恢复并导致不完整的研究输出。选项 D 在恢复策略可能成功时不必要地终止整个工作流。

**问题 9：** 综合智能体验证声明需 2-3 往返增加 40% 延迟。85% 是简单事实检查。如何减少开销？

A) 给综合智能体范围化 verify_fact 工具处理简单查询，复杂验证继续通过协调器
B) 综合智能体批量累积验证需求一次性返回
C) 给综合智能体所有网络搜索工具权限
D) 网络搜索智能体主动缓存额外上下文

**正确答案：A** — 选项 A 应用最小权限原则，仅给综合智能体 85% 常见情况（简单事实验证）所需的能力，同时为复杂情况保留现有的协调模式。选项 B 的批处理方法创建阻塞依赖，因为综合步骤可能依赖早先验证的事实。选项 C 过度配置综合智能体，违反关注点分离。选项 D 依赖推测性缓存，无法可靠预测综合智能体需要验证什么。

### 场景：用于持续集成的 Claude Code

**问题 10：** 你的流水线脚本运行 `claude "Analyze this pull request for security issues"` 但作业无限期挂起。日志显示 Claude Code 正在等待交互输入。在自动化流水线中运行 Claude Code 的正确方法？

A) 添加 -p 标志：`claude -p "Analyze this PR..."`
B) 设置 CLAUDE_HEADLESS=true 环境变量
C) 从 /dev/null 重定向 stdin
D) 添加 --batch 标志

**正确答案：A** — -p（或 --print）标志是运行 Claude Code 非交互模式的文档化方式。它处理提示，将结果输出到 stdout，然后退出而不等待用户输入——正是 CI/CD 流水线所需要的。其他选项引用不存在的功能（CLAUDE_HEADLESS 环境变量、--batch 标志）或使用未正确处理 Claude Code 命令语法的 Unix 变通方案。

**问题 11：** 降低 API 成本：阻塞合并前检查和隔夜技术债务报告。经理提议都用 Batches API（50% 节省）。如何评估？

A) 仅技术债务报告用批处理，合并前检查保持实时
B) 都用批处理加状态轮询
C) 都保持实时避免排序问题
D) 都用批处理加超时回退

**正确答案：A** — 批处理最长 24 小时不适合阻塞检查。B 对阻塞工作流不可接受（"通常更快"不足以保证）。C 是一种误解——批处理结果可通过 custom_id 字段关联，不存在排序问题。D 增加不必要的复杂性。

**问题 12：** 14 文件 PR 单遍审查产生不一致反馈、遗漏 bug 和矛盾标记。如何重构？

A) 按文件单独分析本地问题，然后单独集成遍历检查跨文件数据流
B) 要求开发者拆分为小 PR
C) 切换更大上下文窗口模型
D) 三次独立审查仅标记共同问题

**正确答案：A** — 将审查拆分为专注的遍历直接解决根因：一次处理太多文件时的注意力稀释。逐文件分析确保一致的深度，而单独的集成遍历捕获跨文件问题。选项 B 将负担转嫁给开发者但没有改善系统。选项 C 误解了更大的上下文窗口不能解决注意力质量问题。选项 D 实际上会通过要求对可能只是间歇性捕获的问题达成共识来抑制真实 bug 的检测。

## 附录

### 技术和概念

- **Claude Agent SDK** — 智能体定义、智能体循环、stop_reason 处理、钩子（PostToolUse、工具调用拦截）、Task 工具生成子智能体、allowedTools
- **MCP** — 服务器、工具、资源、isError 标志、工具描述、工具分配、.mcp.json、环境变量展开
- **Claude Code** — CLAUDE.md 层级、.claude/rules/（YAML frontmatter 路径范围）、.claude/commands/、.claude/skills/（SKILL.md frontmatter：context: fork、allowed-tools、argument-hint）、计划模式、直接执行、/memory、/compact、--resume、fork_session、Explore 子智能体
- **Claude Code CLI** — -p / --print、--output-format json、--json-schema
- **Claude API** — tool_use + JSON 模式、tool_choice（"auto"、"any"、强制选择）、stop_reason（"tool_use"、"end_turn"）、max_tokens、系统提示
- **Message Batches API** — 50% 节省、24 小时窗口、custom_id、轮询完成、不支持多轮工具调用
- **JSON Schema** — 必需/可选、枚举、可空、"other" + 详情、严格模式
- **Pydantic** — 模式验证、语义验证错误、验证-重试循环
- **内置工具** — Read、Write、Edit、Bash、Grep、Glob
- **少样本提示** — 歧义场景示例、格式演示、泛化
- **提示链** — 顺序分解为专注遍历
- **上下文窗口管理** — 令牌预算、渐进摘要、迷失在中间、草稿本文件
- **会话管理** — 恢复、fork_session、命名会话、上下文隔离
- **置信度评分** — 字段级置信度、标记验证集校准、分层抽样
